# なまにえ メディア開発計画

## 1. 前提の整理

### 1.1 参照資料の役割
- **namanie-N3-minimal-unease.html** … サイトの**ビジュアル・UX・情報設計**の参照。フォーマットは問わない。
- **2025.html** … **記事の構造・トーン・コンテンツ種別**のサンプル。HTMLはあくまで内容伝達用。

### 1.2 メディアの定位（参照から読み取ったもの）
| 項目 | 内容 |
|------|------|
| 名称 | なまにえ |
| コピー | Raw Offerings from KCA |
| 母体 | KCA（Art × Business School） |
| トーン | 生のまま差し出す、未完を許容、洗練より生々しさ |
| 記事の性格 | 長文エッセイ・文化論・「いま」を読み解く論考（MBTI・自分らしさ・副キャラ・レコメンド等） |
| 読者像 | ビジネスとアートの接点に関心がある層、KCAコミュニティ周辺 |

## 2. メディアとしてどうあるべきか

### 2.1 目的
- **コンテンツの永続的な公開** … 記事はURLで共有・引用され、長く読まれる
- **執筆体験の良さ** … 編集体制・複数執筆者（Writers）を想定したワークフロー
- **デザインの一貫性** … N3「Minimal Unease」の世界観を技術で再現・維持
- **拡張の余地** … 将来の連載・特集・メールマガジン等を考えた設計

### 2.2 必要な機能（要件レベル）
- **公開ページ**: トップ、記事一覧、記事詳細、About、Writers、Contact
- **記事**: タイトル・メタ（著者・日付・読了時間）・本文（見出し・段落・引用・リスト・画像）
- **デザイン**: 指定フォント（Noto Serif JP / Inter）、カラーパレット、余白・タイポグラフィ、ミニマルなアニメーション（ヒーロー、一覧のreveal、赤いドット等）
- **執筆・運用**: 記事の作成・編集・公開日管理、著者管理、画像のアップロードと表示
- **SEO・OGP**: メタ情報、OGP、構造化データ（Article）

### 2.3 形式に縛られない、という意味
- 納品物が「静的HTMLの量産」である必要はない
- **コンテンツはデータ**（Markdown / MDX / CMS）で持ち、**見た目はテンプレート＋スタイル**で表現する
- デザイン参照は「この見え方・この体験」を実現するための仕様であり、実装はReact/Vue/Web Components/SSG/CMS などから最適な組み合わせを選ぶ

## 3. 技術スタック（確定方針）

### 3.1 前提
- **ホスティング**: **Vercel** で行う（確定）。
- **コンテンツ更新**: 記事の追加・更新のたびに**ビルドし直す**運用でよい。将来の **Headless CMS** 導入も見据え、データ層は差し替え可能にしておく。
- **実装の面倒さ**: 判断要素に含めない（実装は AI に依頼する前提のため、手間・ステップ数で技術を狭めない）。

### 3.2 採用スタック
| 項目 | 採用 |
|------|------|
| **ホスティング** | Vercel |
| **コンテンツ** | Markdown / MDX（リポジトリの `content/articles/` 等）。将来 CMS に切り替える場合も、取得したデータを同じスキーマで扱えるようにする。 |
| **フレームワーク** | Astro または Next.js (App Router) のいずれか（デザイン再現・記事ビルドに適した方で選択）。 |
| **スタイル** | Tailwind CSS + CSS Variables（N3 の `--bg`, `--ink`, `--red` 等をそのまま移植）。 |
| **フォント** | Noto Serif JP, Inter（next/font または各 FW の Web フォント）。 |

記事更新 → Git  push → Vercel がビルド・デプロイ、という流れを基本とする。将来 CMS を入れる場合は、ビルド時または On-Demand で CMS API から取得する形に拡張する。

### 3.3 技術選定で決めておくこと
1. **記事の保存形式** … 現時点は Markdown / MDX。スキーマ（frontmatter）は将来の CMS フィールドと対応づけ可能にしておく。
2. **ルーティング** … トップ・一覧・詳細・About・Writers・Contact の URL 設計（例: `/`, `/articles`, `/articles/[slug]`, `/about`, `/writers`, `/contact`）。
3. **画像** … 記事内画像の置き場（`public/articles/` 等）と最適化（next/image, Astro Image 等）。
4. **インタラクション** … 赤いドット・ヒーローアニメ・reveal の再現範囲と実装方法（Vanilla JS / フレームワーク の切り分け）。

## 4. 開発フェーズ案

### Phase 0: 仕様・設計の確定（1〜2週間）
- [ ] フレームワークの決定（Astro または Next.js）
- [ ] 記事スキーマの定義（frontmatter: title, slug, author, date, readMinutes, body。将来 CMS のフィールドと対応可能に）
- [ ] デザイン Tokens の切り出し（色・フォント・スペーシングを 1 箇所で管理）
- [ ] サイトマップ・URL 一覧の確定

### Phase 1: 基盤とデザイン実装（2〜3週間）
- [ ] プロジェクト初期化（Next/Astro 等）、リポジトリ構成
- [ ] デザインシステムの実装（CSS Variables, タイポグラフィ, コンポーネント: Nav, Footer, PageIndicator）
- [ ] トップページ: ヒーロー、セパレータ、記事一覧ブロック（仮データでOK）
- [ ] 記事一覧ページ: 一覧表示、日付・著者・読了時間
- [ ] 共通: レイアウト、フォント読み込み、赤いドット（オプション）

### Phase 2: 記事詳細とコンテンツ（2〜3週間）
- [ ] 記事詳細ページ: 戻るリンク、タイトル、メタ、本文（Markdown/MDX または CMS からレンダリング）
- [ ] 本文スタイル: 段落、見出し（h2/h3）、引用、リスト、画像、drop cap 等（N3 の .article-detail .body を再現）
- [ ] サンプル記事 1 本を 2025.html の内容で投入（Markdown 化して画像パスだけ整備）
- [ ] 読了時間の算出（本文文字数ベースで自動 or frontmatter）

### Phase 3: 固定ページとコンテンツ充実（1〜2週間）
- [ ] About ページ: 文言・3 原則・大きな「贄」とスタンプ
- [ ] Writers ページ: 著者一覧（データは YAML/JSON または CMS）
- [ ] Contact ページ: 文言・リンク・フォーム（送信先は Formspree / Netlify Forms / 自前API のいずれか）
- [ ] 記事を 2〜3 本追加（既存サンプル＋新規）

### Phase 4: 品質・公開準備（1〜2週間）
- [ ] OGP・メタタグ・favicon
- [ ] 構造化データ（JSON-LD: WebSite, Article）
- [ ] パフォーマンス: フォント subset、画像最適化、LCP 確認
- [ ] Vercel 本番デプロイ・ドメイン設定（main マージでデプロイ）

### Phase 5（任意）: 運用強化
- [ ] 管理用の簡易 UI（記事追加・下書き）または CMS 連携の本格化
- [ ] メールマガジン・RSS
- [ ] 検索・タグ・カテゴリ（必要に応じて）

## 5. 記事コンテンツの扱い（2025.html から）

- **タイトル**: 2025年後半という「いま」を読み解き見えてくる世界とは
- **構造**: 見出し（h2）で章立て、本文・リスト・画像（キャプション付き）・補足説明
- **Markdown 化する場合の例**:
  - 見出し → `##`, `###`
  - 画像 → `![alt](path)` または MDX で `<Figure>` コンポーネント
  - 引用・注釈 → blockquote やカスタムブロック
- **画像**: 参照 HTML では `images/image1.png` 等。実装では `public/articles/2025/` などに配置し、パスを統一する方針が扱いやすい。

## 6. まとめ

| 観点 | 方針 |
|------|------|
| 形式 | HTML には縛られない。コンテンツはデータ、見た目はコンポーネント＋デザインシステム。 |
| ホスティング | **Vercel**。記事更新ごとにビルドし直す運用。 |
| コンテンツ | **Markdown/MDX** で開始。データ層は将来の Headless CMS 差し替えを想定した設計。 |
| スタック | Astro または Next.js + Tailwind + CSS Variables。実装の面倒さは判断要素にしない。 |
| デザイン | N3 の色・フォント・余白・コンポーネントを Tokens とコンポーネントで再現。 |
| 記事 | 2025.html のような長文・見出し・画像・リストを Markdown/MDX で表現し、レンダリングで N3 スタイルをあてる。 |
| スケジュール | Phase 0〜4 でおおむね 2〜3 ヶ月。Phase 1 完了で「見た目と一覧」、Phase 2 で「記事が読める」状態を目標にする。 |

この開発計画をベースに、フレームワークの確定と Phase 0 のタスクから着手する。
